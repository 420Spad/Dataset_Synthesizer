17 <a name=summary>
18 # Summary
19 
20 (<a href=#contents>Back to Contents</a>)
21 
22 **DLBroker** is a router for connecting simulations to (possibly
23 remote) processes training neural networks. While it is specifically
24 designed to be used as part of the DLToolkit, in conjunction with the
25 DLEnvironment library and <a href=https://ls-gitswarm.nvidia.com/eleitch/DLTrainer>DLTrainer</a>
26 framework (more below), it is effectively a general-purpose message
27 broker that additionally provides auto-routing logic for messages
28 between specialized deep-learning clients. It is intended to provide
29 a fast, scalable routing service for multiple simulation environments
30 communicating with multiple trainers.
31 
32 This repo provides the DLBroker application, a C++ client library
33 for connecting to the broker, and a Python module built on top of the
34 same C++ client library, providing a single client interface for
35 deep-learning simulations implemented in C++ and neural-network
36 implementations running in Python.
37 
38 DLBroker is designed so that both the broker and client code can be be
39 compiled and run on Linux, BSD, or Windows platforms. Compilation for
40 Windows and Unix variants is described <a href=#compile>below</a>.

44 # Context
45 
46 (<a href=#contents>Back to Contents</a>)
47 
48 The general problem of simulations using reinforcement learning (RL)
49 requires that at each step of the simulation, state data for AI
50 instances can be pushed to a neural network, and that the network
51 return appropriate actions to the AI instances.
52 
53 Traditionally, this requires strong coupling between the simulation
54 generating state data and the RL networks consuming it. This is less
55 than ideal in a research context, because of platform mismatch between
56 data generation (Windows, for gaming sims for example) and
57 neural-network processing (often unix-based), and also because it
58 makes on-the-fly reconfiguration of sims, or rapid iteration in
59 developing network implementations, difficult at best.
60 
61 <br>
62 The DLToolkit includes several components to address these issues:
63 <br>
64 <br>
65 
66 **DLEnvironment** The DLEnvironment library provides a simple layer allowing for
67 decoupling of data generation and processing, by providing a generic
68 remote interface as a C++ DLL that can be integrated with a simulation
69 to provide the capability to configure AI instances on the fly, and
70 that allows the sim to communicate state data and receive actions from
71 a remote training service. The remote framework uses simple JSON configuration
72 files to specify the trainer IP address and port, and the trainer
73 provides the AI configuration when the sim connects.
74 
75 **DLTrainer** The DLTrainer codebase provides a general-purpose Python
76 framework for receiving and processing data from multiple sim
77 environments, and batching and routing the data between multiple
78 network implementations. The framework allows network implementations
79 to be written as standalone modules and deployed within the framework,
80 provided they adhere to a simple API. With the remote framework
81 provided by the DLEnvironment library, the trainer can be stopped and
82 restarted and any running sims will automatically reconnect.
83 
84 **DLBroker** The combination of DLEnvironment and DLTrainer provides a
85 generic _many-to-one_ architecture, allowing many sim environments to
86 send data simultaneously to a single trainer (possibly running
87 multiple network implementations). DLBroker extends this architecture
88 to support _many-to-many_ connections. Trainers and sim environments
89 connect to the broker instead of each other, and the broker handles
90 auto-routing between data generators (sim environments) and data consumers
91 (trainers).
92 
93 * This allows many instances of a sim environment to be run without
94  needing separate configuration files for each instance (all
95  environments connect to the same broker, instead of to separate
96  trainers)
97 
98 * Allows the broker to automatically handle routing of data between
99  simulation environments and the trainers that can consume them.
100 
101 * Allows the broker to balance data distribution among connected
102  trainers.
103 
104 * Allows sim data to be forwarded to multiple trainers (i.e., for
105  supervised learning applications, or separate control of AI
106  instances by different servers)
107

116 # Broker Usage
117 
118 Currently DLBroker provides a simple executable that is invoked with the following optional parameters:
119 
120 `DLBroker [port=9505] [verbose=f] [highwatermark="0B"] [lowwatermark="0B"] [nthread=10]`
121 
122 (defaults are shown above).
123 
124 Parameters are:
125 
126 * `port` The port on which the broker should listen, which defaults to 9505.
127 
128 * `verbose` Set to true to print more information about what the broker is doing.
129 
130 * `highwatermark` This is a desired cap on the broker's memory usage.
131  If specified, the broker will attempt to limit instantaneous memory
132  consumption to the specified limit. To make this more
133  human-friendly to specify, it expects a string like "500 kB",
134  "100MB", or "1.2 GB". Be careful -- I pedantically distinguish
135  between "b" (bits) and "B" (bytes). When a non-zero value is given
136  for the high-water mark, the broker will throttle communications if
137  internal memory consumption reaches this limit.
138 
139 * `lowwatermark` To be used in conjunction with the high-water mark.
140  If the broker throttles comms at the high-water mark, they will be
141  unthrottled when the memory consumption drops to the low-water mark.
142  This provides some hysteresis, preventing the system from constantly
143  throttling/unthrottling at the high-water mark as individual
144  messages are consumed. If not specified, the broker defaults to 80%
145  of the high-water mark.
146 
147 * `nthread` This determines how many worker threads the broker master
148  spawns to handle requests. You generally won't care about this, but
149  on a CPU-bound multi-core system, this can be used to optimize
150  throughput by allowing the system scheduler to deploy worker threads
151  on independent cores.
152 
153 Once deployed, the broker listens for connection requests from broker
154 clients, and processes messages passed between connected clients.
155 Supported clients and associated behaviors are detailed below.

159 # Client Libraries and Usage
160 
161 (<a href=#contents>Back to Contents</a>)
162 
163 By default DLBroker supports the following client types and behaviors:
164 
165 * **BrokerClient** A generic broker client, which can publish data to
166  arbitrary channels, and subscribe to arbitrary channels. Maps to
167  C++ and Python `BrokerClient` classes.
168 
169 * **AutoBrokerClient** A generic broker client, to which the broker
170  automatically assigns publish and reply channels. Maps to C++ and
171  Python `AutoBrokerClient` classes.
172 
173 * **DLEnvironmentBrokerClient** A specialized type of AutoBrokerClient
174  for use with the DLEnvironment library. This client advertises a
175  simulation type to the broker, and has channels automatically
176  assigned when it connects. It supports a set of DL-specific
177  messages (see below), and messages sent to the broker by this client
178  are automatically routed to appropriate trainers (see <a href=#dlenvusage>usage description</a>
179  below), by simulation type. Maps to C++ and Python
180  `DLEnvironmentBrokerClient` classes.
181 
182 * **DLTrainerBrokerClient** A specialized type of AutoBrokerClient for
183  use with the DLTrainer RL service. This client advertises a list of
184  simulation types it can process when it connects to the broker.
185  When clients of type DLEnvironmentBrokerClient connect to the
186  broker, messages from them are auto-routed to DLTrainerBrokerClient
187  clients, matched by simulation type. Maps to C++ and Python
188  `DLTrainerBrokerClient` classes.`
189 
190 All supported clients will automatically connect to the broker, and
191 re-establish a connection to the broker if a network partion occurs,
192 or if the broker process is restarted.

194 <a name=clientusage>
195 ## Client Usage Examples
196 
197 ### BrokerClient
198 
199 The generic broker client acts like a traditional message-broker
200 client. It supports simple `subscribe` and `publish` methods:
201 
202 <br>
203 **C++**
204 
205 The entry point to the DLL is via the BrokerUtils.h header. Clients
206 should be instantiated anad destroyed via the corresponding factory
207 methods in BrokerUtils:
208 
209 ```cpp
210  using namespace DLBroker;
211  BrokerClient* client = BrokerUtils::newBrokerClient();
212 ```
213 
214 The client APIs are documented in the headers for the coresponding
215 clases. For example, to publish a message using a BrokerClient, use
216 the `publish` method:
217 
218 ```cpp
219  client->publish("mychannel", "test message");
220 ```
221 
222 or
223 
224 
225 ```cpp
226  client->publish("mychannel", "test message", size, bytePtr);
227 ```
228 
229 for binary message content, where `bytePtr` is a pointer to raw binary
230 data.
231 
232 A typical usage pattern is to subscribe to a set of channels, and
233 register a handler to be called whenever a message is read from the
234 broker:
235 
236 
237 ```cpp
238  client->registerReadHandler(handlerFn, handlerArgs);
239  client->subscribe("mychannel");
240  client->subscribe("anotherchannel");
241 ```
242 
243 In the example above, `handlerFn` would be called with `handlerArgs`
244 as the last argument whenever a message is received on channels
245 `mychannel` or `anotherchannel`. (See `BrokerTypes.h` included from
246 `BrokerUtils.h` for the declaration of `handlerFn`).
247 
248 <br>
249 **Python**
250 
251 The Python module `pyBrokerClient` supplies a similar
252 interface to the client library:
253 
254 
255 ```python
256  >>> import pyBrokerClient as pbc
257  >>> client = pbc.BrokerClient()
258  >>> client.publish("mychannel", "test message")
259 ```
260 
261 The example below subscribes to channel `mychannel` and calls the
262 `print()` function whenever a message is received.
263 
264 
265 ```python
266  >>> client.registerReadHandler(print, None)
267  >>> client.subscribe('mychannel')
268 ```
269 
270 Any Python method invoked with incorrect arguments will print a synopsis of expected
271 arguments. For example:
272 
273 ```python
274  >>> client.registerReadHandler()
275  ...
276  ValueError:
277 
278  Usage: registerReadHandler(handlerFn, handlerArgs)
279 
280  where handlerFn should be handlerFn(channel, lastReadStr, handlerArgs)
281 
282  >>>
283 ```

286 ### DLEnvironmentBrokerClient
287 
288 In general, DL environments and trainers support a very simple
289 communications loop, based on the following logic:
290 
291 * A sim environment can send a healthcheck request to get status
292  information about whether a trainer is present to process its data.
293 
294 * Each cycle of a simulation is referred to as an _experiment_, and
295  each experiment consists of _episodes_, which can consist of an
296  arbitrary number of simulation _ticks_.
297 
298 * The trainer controls how many ticks constitute an episode, and how
299  many episodes constitute an experiment.
300 
301 * When the trainer communicates that an episode has ended, the sim
302  environment is expected to start a new episode, and when the trainer
303  communicates that an experiment has ended, the environment is
304  expected to start a new experiment.
305 
306 <br>
307 **C++**
308 
309 A new DLEnvironment client is instantiated via a factory method in BrokerUtils:
310 
311 ```cpp
312  using namespace DLBroker;
313  DLEnvironmentBrokerClient* client = BrokerUtils::newDLEnvironmentBrokerClient("SimpleTrainingApp");
314 ```
315 
316 DLEnvironmentBrokerClient supports methods for each of the logical steps listed above:
317 
318 ```cpp
319  std::string resp;
320  if(client->healthcheck(resp) == SocketConnection::READ_DONE) {
321 
322  // Check response to see if servers are present
323 
324  if(serversArePresent(resp)) {
325 
326  client->experimentstart(resp);
327  unsigned expId = getExperimentId(resp);
328 
329  client->episodestart(formatEpisodeStartData(expId), resp);
330  client->tick(formatTickData(expId), resp);
331 
332  processActions(resp);
333  ...
334  }
335  }
336 ```

339 ### DLTrainerBrokerClient
340 
341 <br>
342 **Python**
343 
344 A new DLTrainer client is instantiated from the `pyBrokerClient` module. For example:
345 
346 ```python
347  >>> import pyBrokerClient as pbc
348  >>> client = pbc.DLTrainerBrokerClient(['SimpleApp', 'AleApp'])
349 ```
350 
351 instantiates a DLTrainer client that tells the broker it can process
352 data from simulation types `SimpleApp` and 'AleApp`. DlEnvironment
353 clients that produce data of those types will be auto-routed by the
354 broker to this client.

402 ## Configuring DLEnvironment and DLTrainer to use the broker
403 
404 The following example walks through some of the steps required to use
405 the broker with the DLEnvironment and DLTrainer codebase. It assumes that the
406 broker executable is running on the localhost, listening on port 9505.
407 
408 Suppose you have a simulation `SimpleTrainingApp` that is compiled
409 with DLEnvironment support. Let' say the executable directory for
410 `SimpleTrainingApp` is located at `$stadir`, which also contains a
411 subdirectory `$stadir/Configuration` where it looks for local configuration
412 files.
413 
414 Suppose that `DLTrainer` is installed in `$dltrainerdir`, which contains
415 JSON config files under directory `$dltrainerdir/config`, and network implementation
416 modules are located somewhere in your PYTHONPATH.
417 
418 We want to control `SimpleTrainingApp` from a neural network running
419 in `DLTrainer`, via the DLBroker executable. To do this, we require only
420 that both be told what IP address and port to use for communication
421 with the broker.
422 
423 We accomplish this for the trainer by creating a config file (say
424 `$dltrainerdir/config/DLTrainer/Trainer_Config.json`) like:
425 
426 ```
427 {
428  "RL" :
429  {
430  "BrokerAddress" : "localhost:9505",
431 
432  "Modules" : ["../simpleTrainingApp/Trainer_Config.json"]
433  }
434 }
435 ```
436 
437 The `BrokerAddress` keyword tells `DLTrainer` the IP address and port
438 to use for communication with the broker, for use with the RL service
439 of the DLTrainer. (See DLTrainer documentation for more details on
440 configuring the trainer services).
441 
442 We start the trainer like (powershell):
443 
444 ```
445 & python DLTrainer.py trainerconfig=$dltrainerdir/config/DLTrainer/Trainer_Config.json
446 ```
447 
448 This tells the trainer to use
449 `$dltrainerdir/config/DLTrainer/Trainer_Config.json` as its config.
450 
451 For the `SimpleTrainingApp`, we modify its configuration
452 `$stadir/Configuration/Remote_Config.json` to read:
453 
454 ```
455 {
456  "BrokerAddress": "127.0.0.1:9505",
457  "SimType": "SimpleTrainingApp"
458 }
459 ```
460 
461 The first line tells the DLEnvironment interface the IP address and port to
462 use for communication with the broker, and the second line is the tag
463 that the broker will use to identify the type of simulation data that
464 we are sending. If it matches one of the tags registered by a
465 `DLTrainer` instance connected to the broker, data for this sim will be
466 automatically routed to that trainer.
467 
468 We start the app like (powershell):
469 
470 ```
471  & cd $stadir; ./SimpleTrainingApp.exe Configuration Local_Config.json
472 ```
473 
474 (where `$stadir/Configuration/Local_Config.json` is a local
475 configuration file that would be used if not configured to talk to a
476 remote trainer (i.e., if `Remote_Config.json` is empty))
477 
478 And that's it!
479 
480 The app should now be talking to the trainer. If multiple instances of
481 the app are started, they will all be connected by the broker to the
482 same trainer process. If multiple `DLTrainer` processes are started,
483 the broker will distribute data from all connected instances of
484 `SimpleTrainingApp` evenly among the trainer processes.
485 
486 New instances of `SimpleTrainingApp` can be attached to the broker,
487 and their data will be immediately routed to attached trainers.
488 Likewise, new `DLTrainer` instances can be attached, and they will
489 begin processing data.
490 
491 If the broker is restarted, the `SimpleTrainingApp` and `DLTrainer`
492 processes will reattach to the broker and resume communication.

495 ## Client callbacks in Python
496 
497 The broker also supports general pub/sub clients. The following
498 example demonstrates use of the broker to spawn powershell processes
499 remotely from Python. This example assumes that the broker executable
500 is running on the localhost on port 9505.
501 
502 ```python
503 import pyBrokerClient as pbc
504 import subprocess
505 
506 def powershellCallback(channel, msg, args):
507  subprocess.call(["C:\\WINDOWS\\system32\\WindowsPowerShell\\v1.0\\powershell.exe", msg])
508 
509 client = pbc.BrokerClient("127.0.0.1", 9505)
510 client.registerReadHandler(powershellCallback, None)
511 client.subscribe("pscmd")
512 ```
513 
514 Sending messages to the channel `pscmd` will spawn the corresponding
515 powershell process on the host machine.
516 
517 The read handler is registered on the client (Python) side, so will
518 remain in effect across restarts of the broker. But the subscribe
519 command is a command to the broker -- that state is therefore lost if
520 the broker is restarted.
521 
522 You can preserve state across broker restarts by placing stateful
523 commands in a connection callback function -- this will be called by
524 the client whenever a connection is established to the broker. For
525 example, adding the following code to the example above would ensure
526 that the client subscribes to the pscmd channel whenever a connection
527 to the broker is re-established:
528 
529 ```python
530 def connectCallback(client):
531  client.subscribe("pscmd")
532 
533 client.registerConnectHandler(connectCallback, client)
534 ```
