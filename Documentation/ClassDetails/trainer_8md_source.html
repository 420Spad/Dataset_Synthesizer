33 <a name=summary></a>
34 # Summary
35 
36 (<a href=#contents>Back to Contents</a>)
37 
38 **DLTrainer** is a generalized Python framework for deploying neural
39 networks. It currently supports two services: one for reinforcement
40 learning (RL), and one for supervised learning (SL). Note that
41 DLTrainer is a _deployment_ environment -- it is not a platform for
42 _developing_ neural networks, and as such is not dependent on any
43 particular development platform, such as pytorch or tensorflow. You
44 can develop using your favorite platform, and deploy it within
45 DLTrainer, as long as your Python environment is set up to support
46 your platform.
47 
48 **The RL service** provides a simple framework for deploying Python
49 reinforcement-learning networks against training data from (possibly
50 remote) DLEnvironment sims. Neural-network implementations can be
51 written as stand-alone Python modules; the trainer automatically
52 handles communication with DLEnvironment sims, as well as batching and
53 routing of DLEnvironment data to the user-supplied network
54 implementations.
55 
56 DLTrainer supports communication with DLEnvironment via the DLBroker
57 router. The broker allows multiple DLEnvironment sims to be routed to
58 multiple copies of DLTrainer, as shown schematically below.
59 
60 **The SL service** provides a simple framework for configurable and
61 reproducible deployment of supervised-learning networks. The service
62 allows parallel deployment of multiple independent SL experiments, or
63 multiple copies of the same SL network with different configurations,
64 or both, and captures the scripts, configuration and output of the
65 training sessions. Just as with the RL service, the SL service of the
66 trainer is DL-platform-agnostic (i.e., pytorch, tensorflow, etc); SL
67 network implementations must only support configuration via
68 command-line arguments, or provide a lodable training module run
69 method that supports passing a configuration dictionary. The SL
70 service is entirely configurable via a JSON hierarchy, described in
71 more detail below.
72 
73 NB: As with any Python environment, modules used by your network
74 implementations will have to be discoverable at runtime. You can add
75 the relevant directories to your PYTHONPATH variable, or via the
76 DLTrainer JSON configuration files (see below). For the RL service,
77 which deploys networks as threads of the main applications, these
78 paths must be available to the DLTrainer runtime environment. For the
79 SL service, which spawns networks as python subprocesses, they can be
80 configured via your environment, or via per-module configurable
81 additions specified in the JSON configuration.

89 <a name=rl_context>
90 ## The RL Problem
91 
92 The general problem of simulations using reinforcement learning (RL)
93 requires that at each step of the simulation, state data for AI
94 instances can be pushed to a neural network, and that the network
95 return appropriate actions to the AI instances.
96 
97 Traditionally, this requires strong coupling between the simulation
98 generating state data and the RL networks consuming it. This is less
99 than ideal in a research context, because of platform mismatch between
100 data generation (Windows, for gaming sims for example) and
101 neural-network processing (often unix-based), and also because it
102 makes on-the-fly reconfiguration of sims, or rapid iteration in
103 developing network implementations, difficult at best.
104 
105 <br>
106 The DLToolkit includes several components to address these issues:
107 <br>
108 <br>
109 
110 **DLEnvironment** The DLEnvironment library provides a simple layer
111 allowing for decoupling of data generation and processing, by
112 providing a generic remote interface as a C++ DLL that can be
113 integrated with a simulation to provide the capability to configure AI
114 instances on the fly, and that allows the sim to communicate state
115 data and receive actions from a remote training service. The remote
116 framework uses simple JSON configuration files to specify the trainer
117 IP address and port, and the trainer provides the AI configuration
118 when the sim connects.
119 
120 **DLTrainer** The DLTrainer codebase provides a general-purpose Python
121 framework for receiving and processing data from multiple sim
122 environments, and batching and routing the data between multiple
123 network implementations. The framework allows network implementations
124 to be written as standalone modules and deployed within the framework,
125 provided they adhere to a simple API. With the remote framework
126 provided by the DLEnvironment library, the trainer can be stopped and
127 restarted and any running sims will automatically reconnect.
128 
129 **DLBroker** The combination of DLEnvironment and DLTrainer provides a
130 generic _many-to-one_ architecture, allowing many sim environments to
131 send data simultaneously to a single trainer (possibly running
132 multiple network implementations). DLBroker extends this architecture
133 to support _many-to-many_ connections. Trainers and sim environments
134 connect to the broker instead of each other, and the broker handles
135 auto-routing between data generators (sim environments) and data consumers
136 (trainers).
137 
138 * This allows many instances of a sim environment to be run without
139  needing separate configuration files for each instance (all
140  environments connect to the same broker, instead of to separate
141  trainers)
142 
143 * Allows the broker to automatically handle routing of data between
144  simulation environments and the trainers that can consume them.
145 
146 * Allows the broker to balance data distribution among connected
147  trainers.
148 
149 * Allows sim data to be forwarded to multiple trainers (i.e., for
150  supervised learning applications, or separate control of AI
151  instances by different servers)
152

160 ## The SL Problem
161 
162 Ease of deployment and repeatability are common concerns among the
163 research groups we've talked to. Many SL training modules are driven
164 by command-line configuration which is not easily retrievable or
165 reproducible once the run is finished, variation of configurations
166 generally requires significant manual intervention, and deployments
167 are not easily automated or scalable.
168 
169 The DLTrainer SL service attempts to improve this situation by
170 providing a simple-to-use JSON front-end for configuration of all SL
171 modules, with a parameter hierarchy that makes it easy to deploy
172 multiple copies of a network with variations of a base configuration.
173 
174 The service also attempts to mitigate the difficulty of managing
175 training sessions by capturing output and errors in a self-contained,
176 timestamped directory structure that mirrors the configuration of the
177 SL service (see below). Each training session deployed in DLTrainer
178 creates new entries that preserve the results of previous runs, and
179 include the configuration and self-contained scripts which can
180 be used to exactly reproduce the session, if desired.
181 
182 The service includes a configurable endtime for each deployed network,
183 so that no manual intervention is required to perform, for example, a
184 battery of fixed-length tests.
185 
186 The service also includes a built in scheduler, so that a large set of
187 fixed-length network variations can be configured at once and deployed
188 on a single node, and the trainer will start each queued network as
189 resources become available, including automatic detection of GPU
190 resource availability.
191 
192 Next steps may include
193 
194 * metrics monitoring and metrics-based session management (abort, restart, iteration)
195 * remote configuration and control of SL training sessions deployed in DLTrainer via the DLBroker
196 * dataset management and automatic routing of remote datasets to SL training sessions via the DLBroker

204 ## Overview
205 
206 The trainer ultimately requires a single JSON file to configure the RL
207 and SL services. When invoked with no arguments, DLTrainer loads the
208 trainer config file specified by the default **TrainerConfig** tag
209 (see <a href=#config_defaults>Parameter Defaults</a>).
210 
211 You can also specify the configuration file by using the
212 `trainerconfig` commandline argument, like:
213 
214 ```shell
215 python DLTrainer.py trainerconfig="path/to/my/config_file.json"
216 ```
217 
218 The philosophy behind configuring DLTrainer is that you should have to
219 write as little JSON as possible to use it. The trainer therefore
220 keeps its own set of default values for all required parameters; you
221 can optionally override any of these parameters in your trainer config
222 file, or omit them and the trainer will use the default values.
223 
224 * Notes on parameter defaults are given in <a href=#config_defaults>Parameter Defaults</a>.
225 
226 * The general structure of the trainer config file is discussed in <a
227 href=#config_trainer>Trainer Configuration File</a>.
228 
229 * Details about configuring the SL service are given in <a href=#config_sl_service>Using the SL Service</a>.
230 
231 * Details about configuring the RL service are given in <a href=#config_rl_service>Using the RL Service</a>.
232 
233 <a name=config_semantics></a>
234 ## Special JSON-Parsing Semantics
235 
236 The DLTrainer JSON parser provides some non-standard functionality:
237 
238 * **Recursive Substitution**
239 
240  Any JSON tag can point to a block of valid JSON, or equivalently
241  to a JSON file containing that block. If the trainer detects a
242  '.json' file extension, it will (recursively) insert the
243  corresponding JSON into the final configuration.
244 
245 * **Path Expansion**
246 
247  Any JSON tag that is detectably a file or path name will be
248  expanded. Detectable paths include filenames with ".json"
249  extensions, or any tag including the name "path" (e.g.,
250  "OutputPath", "AddPath"). Either absolute or relative paths
251  (i.e., "../path/to/some/file") are acceptable, and environment
252  variable substitution is supported on all platforms. Relative
253  paths are always expanded relative to the directory of the
254  JSON file containing the tag.
255 
256 * **Comments**
257 
258  The DLTrainer JSON parser treats any line preceded with a '#'
259  character as a comment. These lines are stripped out before
260  parsing.
261 
262 * **AddPath Directive**
263 
264  The tag **AddPath**, pointing to a path or a list of paths, has
265  several meanings, depending on where it occurs:
266 
267  At the top-level in the trainer config file, it causes the trainer
268  to add the specified paths to its system path (equivalent to
269  adding them to your PYTHONPATH environment variable), which will
270  affect all subsequent resolution of python modules referenced in
271  the configuration.
272 
273  In an SL module configuration, it causes those paths to be added
274  to the system path only for the spawned subprocesses corresponding
275  to that SL module. Thus, modules can reference the same python
276  module name in different contexts, without namespace conflicts
277  (for example, many experiments start their networks with modules
278  called 'training.py'. This allows each module to specify _which_
279  training.py it means).
280 
281 * **Include Directive**
282 
283  The tag **Include** is the equivalent of direct JSON substitution
284  when encountered by the trainer. I.e., if an `"Include"` tag points
285  to a JSON file, it is equivalent to inserting the contents of that
286  file at the current JSON block level. This allows options used by
287  multiple JSON files to be placed in a separate file and simply
288  included where desired.

290 <a name=config_defaults></a>
291 ## Parameter Defaults
292 
293 Default values for all services are _always_ taken from the file
294 `config/DLTrainer/Parameter_Defaults.json`. You can edit this file
295 directly if you wish to change default behaviors, or you can simply
296 override them in your trainer config file. To restore this file to
297 program defaults, simply remove it, and it will be re-created the next
298 time you run DLTrainer.
299 
300 If the file cannot be created, the trainer will simply initialize
301 parameters internally, but you will not be able to modify the default
302 behaviors globally.
303 
304 The structure of this file mirrors the service hierarchy of DLTrainer
305 itself, at the highest level containing default parameters used to
306 configure the DLTrainer process itself, and sub-blocks labeled "RL"
307 and "SL", containing default parameters for the RL and SL services,
308 respectively.. Defaults for the services can be overridden for any
309 module included in the service, described in more detail below.
310 
311 For example, Parameter_Defaults.json might look something like this:
312 
313 ```javascript
314 {
315  #------------------------------------------------------------
316  # Parameters for DLTrainer
317  #------------------------------------------------------------
318 
319  # True to print trainer config JSON on startup
320 
321  "ConfigPrint" : false,
322 
323  # Name of the trainer configuration file
324 
325  "TrainerConfig" : "Trainer_Config.json",
326 
327  ...,
328 
329  "RL" :
330  {
331  #------------------------------------------------------------
332  # Parameters for RL
333  #------------------------------------------------------------
334 
335  # Broker IP Address : port
336 
337  "BrokerAddress" : "127.0.0.1:9505",
338 
339  # Local HTTP Server port
340 
341  "HTTPServerPort" : 8505,
342 
343  ...
344 
345  },
346 
347  "SL" :
348  {
349  #------------------------------------------------------------
350  # Parameters for SL
351  #------------------------------------------------------------
352 
353  # Name of the directory under which to capture SL output (no path will default to system tempdir)
354 
355  "OutputPath" : null,
356 
357  # True if SL networks require GPUs (used to determine if the trainer should wait for resource availability)
358 
359  "RequiresGpu" : false,
360 
361  ...
362 
363  }
364 
365 }
366 ```
367 
368 <a name=config_trainer>
369 ## Trainer Configuration File
370 
371 Your trainer configuration file should have the same structure as
372 Parameter_Defaults.json. On startup, your configuration file is
373 merged with Parameter_Defaults.json to produce the trainer's final
374 configuration. Parameters at the highest level override defaults for
375 the trainer process itself, and parameters in the "RL" and "SL" blocks
376 override default parameters for those services.
377 
378 For example, if your trainer config file contained the blocks:
379 
380 ```javascript
381 {
382  "ConfigPrint" : true,
383 
384  "RL" : {
385  "BrokerAddress" : "192.168.0.1:9606",
386  "Modules" : ["../simpleTrainingApp/Trainer_Config.json", "../aleApp/Dqn_Config.json"]
387  },
388 
389  "SL" : {
390  "OutputPath" : "$DEFAULT_SL_OUTPUT_PATH",
391  "Modules" : ["../DLTest/Trainer_Config.json", "../DLAA/Reference/Trainer_Config.json"]
392  }
393 
394 }
395 ```
396 
397 then the value of "ConfigPrint" would override the trainer's value
398 from Parameter_Defaults.json above, broker address `192.168.0.1:9606`
399 would be used instead of `127.0.0.1:9505` for the RL service, and
400 environment variable `$DEFAULT_SL_OUTPUT_PATH` would be used as the
401 default output path for all SL modules.
402 
403 Each service block in your trainer config file is expected to contain
404 a list of modules (i.e., experiment type) that that service will
405 deploy. These can be lists of valid JSON blocks, or lists of files
406 containing JSON for those modules. Each entry defines the complete
407 configuration for a specific network type that the service will
408 deploy. Each entry in the list inherits the parameter values of the
409 enclosing block (so for example, each SL module above would inherit
410 `"OutputPath" : "$DEFAULT_SL_OUTPUT_PATH"`), and any inherited parameter can
411 be overridden in the configuration for each module.
412 
413 The module configuration is described for each service below, in <a
414 href=#config_sl_service>Using the SL Service</a>, and <a
415 href=#config_rl_service>Using the RL Service</a>, respectively.
416 
417 Omitting the block for any service will simply cause that service not
418 to be started.
419 
420 _NB: any configuration parameter can also be overridden on the commandline
421 using the same namespace hierarchy (case-insensitive). For example:_
422 
423 
424 ```shell
425 python DLTrainer.py configprint=true rl.brokeraddress="192.168.0.1:9606" sl.outputpath="$DEFAULT_SL_OUTPUT_PATH"
426 ```
427 
428 _would also set these parameters._

437 ## SL Service JSON Configuration
438 
439 As described in <a href=#config_trainer>Trainer Configuration
440 File</a>, the SL service configuration includes a `"Modules"` tag,
441 consisting of a list of modules (experiments) that the service wil
442 spawn. Each entry in the `"Modules"` list should be a JSON block (or
443 a file containing a JSON block) that includes _at least_ the following
444 tags (for example, the following might be the contents of
445 `../DLAA/Reference/Trainer_Config.json`, referenced above):
446 
447 ```javascript
448 {
449  # These are all DLTrainer-specific arguments
450 
451  "Name" : "DLAA",
452  "Impl" : "imgrst_recurrent_DLTrainer"
453 
454  # Anything in the Config block is specific to the implementation
455  # (i.e., passed to the implementation, and not used by the trainer)
456 
457  "Config" : {
458  "gpu" : 0
459  }
460 }
461 ```
462 
463 * The **Name** tag is used to identify the module for logging,
464  constructing output paths, etc.
465 
466  The **Name** tag is also used to provide support for command-line
467  manipulation of parameters for your module. For example:
468 
469  ```shell
470  python DLTrainer.py sl.dlaa.impl="someOtherModule"
471  ```
472 
473  would override the value of the "Impl" tag in the configuration above, while
474 
475  ```shell
476  python DLTrainer.py sl.dlaa.config.gpu=2
477  ```
478 
479  would override the value of the "gpu" tag inside the "Config" block.
480 
481 * The **Impl** tag should point to your custom top-level python module
482  used to start networks of this type. The configuration block above
483  says that your network should be started by calling the 'run' method
484  of module imgrst_recurrent_DLTrainer.py. (see <a
485  href=#config_sl_python>SL Network Implementations</a> for a
486  description of what DLTrainer expects from your network implementation).
487 
488 * If present, the **Config** block contains any network
489  implementation-specific configuration. These values will be passed
490  by the trainer to your implementation as command-line arguments, or
491  a config dictionary, depending on how your process is configured
492  (see below)
493 
494 * Additionally, if the **ExeType** tag is present, it allows you to
495  control how your module is launched by the trainer. The following
496  value are supported:
497 
498  * "python_cmdline" -- this is the default. Your implemention is
499  treated as a python main, and the trainer passes the config
500  parameters as a set of command-line switches to that process
501 
502  * "python_loadable" -- Your implementation is treated as a
503  loadable python module exporting a `run(config)` method (see
504  below). The trainer will load it and passthe configuration in
505  as a python dictionary.
506 
507  * "other" -- Not a python script, but an arbitrary binary. The
508  trainer calls the executable and passes the config parameters as
509  a set of command-line switches to that process. See <a
510  href=#config_sl_python>SL Network Implementations</a> below.
511 
512  For the commend-line versions, a set of additional JSON keys
513  allows you to control how your config is turned into
514  command-line argunents.
515 
516 * In addition to these tags, DLTrainer supports a growing list of
517  parameters for configuring output, deployment control, etc., of your
518  module's networks. See the file `Parameter_Defaults.json` for a
519  full list and inline documentation.
520 
521 These parameters are used by the trainer when spawning your network
522 process. Any _other_ parameters you specify in the block for your
523 module will be passed to your module as a python dictionary on startup
524 (see <a href=#config_sl_python>SL Service Network Implementations</a>
525 below).
526 
527 As shown, this example configuration would cause a single copy of a
528 DLAA network to be run by the SL service. However, if your
529 configuration contains the **Networks** tag, a network for each entry in
530 that list will be run. For example:
531 
532 ```javascript
533 {
534  # These are all DLTrainer-specific arguments
535 
536  "Name" : "DLAA",
537  "Impl" : "imgrst_recurrent_DLTrainer",
538 
539  # These are user-defined (in this case DLAA-specific) arguments
540 
541  "Config" : {
542  "Include" : "Cmdline_Defaults.json",
543  "DLAAConfigs" : "Configs.json",
544  "ActiveConfig" : "msalvi-dgx-wrae"
545  },
546 
547  # Networks to run. These inherit all parameters above, and allow
548  # you to override selected parameter values: trainer service
549  # parameters are accessed directly, and "Config" keywords above
550  # can be modified by reopening that namespace
551 
552  "Networks" : [{"Config" : {"gpu" : 0, "CROP_SIZE" : 128, "BATCH_SIZE" : 2}},
553  {"Config" : {"gpu" : 1, "CROP_SIZE" : 128, "BATCH_SIZE" : 4}},
554  {"Config" : {"gpu" : 2, "CROP_SIZE" : 128, "BATCH_SIZE" : 8}},
555  {"Config" : {"gpu" : 3, "CROP_SIZE" : 192, "BATCH_SIZE" : 2}}]
556 }
557 ```
558 
559 would cause the SL service to run four networks simultaneously. Just
560 like the "Modules" list in the trainer's config, each network in the
561 "Networks" list inherits all parameters from its parent block, so you
562 can just override parameters of interest. Trainer SL Service
563 parameters are specified directly, and implementation-specific
564 parameters are specified by reopening the "Config" namespace. In the
565 example above, each network inherits everything in the parent "Config"
566 block, and only varies "gpu", "CROP_SIZE", and "BATCH_SIZE".
567 
568 _NB: And just as each module's parameters can be overridden on the
569 commandline, you can override any network's parameters too. For example:_
570 
571 ```
572 python DLTrainer.py sl.dlaa.network_0.config.crop_size=192
573 ```
574 
575 _would override the network specification above._
576 
577 Two additional parameters control how the python network
578 implementation is run:
579 
580 * "AddPath"
581 
582  This can be a path, or a list of paths.
583 
584  If using "ExeType" : "python_cmdline" or "python_loadble", the
585  trainer will add these to the python path (equivalent to the
586  PYTHONPATH) for the spawned subprocess. This allows you to invoke
587  implementations outside of the trainer's system path, and to
588  re-use the same implementation name (e.g., the ubiquitous
589  'training.py') for different modules.
590 
591  If using "ExeType" : "other", the trainer will add these to the
592  system path for the spawned process (equivalent to the PATH
593  variable).
594 
595 * "PyPath"
596 
597  If specified, the directory in which to execute your network.
598  Some training implementations assume that the code is being
599  executed in a standard place (i.e., to look for data files, etc).
600  This is not in general the case when run from the trainer. But if
601  you specify "PyPath", the trainer script will explicitly chdir to
602  that directory before running your implementation.
603 
604  NB: This is not in general the same as auto-detecting the path in
605  which your top-level implementation resides, which is why I leave
606  it as a user-definable parameter.
607 
608 <a name="config_sl_python"></a>
609 ## SL Service Network Implementations
610 
611 **Command-line Python Modules**
612 
613 (`"ExeType" : "python_cmdline"`)
614 
615 In this case, the trainer assumes your implementation specified in the
616 "Impl" tag is a python command line (invoked like `python training.py
617 --boolswitch --param1 val1 --param2 val2`). The trainer will take
618 parameters from your network JSON "Config" block and call your
619 implementation with the equivalent command line. Construction of the
620 command line is controlled by the "CmdlinePrefix" and "CmdlineSep"
621 keywords.
622 
623 **Loadable Python Modules**
624 
625 (`"ExeType" : "python_loadable"`)
626 
627 In this case or use with DLTrainer, your module (the python code
628 pointed to in the "Impl" tag for your module) must be a loadable
629 python module (i.e., not a python 'main') that exports a run method
630 with the signature:
631 
632 ```python
633 def run(config):
634  # Start a network, using parameters passed in the config dictionary
635  ...
636 ```
637 
638 DLTrainer wraps this method in its own script to deploy networks, and
639 passes in a dictionary of custom parameter values specified in the
640 JSON above.
641 
642 For example:
643 
644 ```javascript
645 {
646  # These are all DLTrainer-specific arguments
647 
648  "Name" : "DLAA",
649  "Impl" : "imgrst_recurrent_DLTrainer",
650 
651  # These are user-defined (in this case DLAA-specific) arguments
652 
653  "Config" : {
654  "Include" : "Cmdline_Defaults.json",
655  "DLAAConfigs" : "Configs.json",
656  "ActiveConfig" : "msalvi-dgx-wrae"
657  }
658 }

661 would cause a python dictionary containing keys **DLAAConfigs**,
662 **ActiveConfig**, and any key defined in `Cmdline_Defaults.json` to
663 the run method of `imgrst_recurrent_DLTrainer.py` when starting DLAA
664 networks (see <a href=#config_semantics>Special Semantics</a> for a

667 **Other Executable**
668 
669 (`"ExeType" : "other"`)
670 
671 In this case, the trainer assumes that the implementation pointed to
672 by the "Impl" keyword is an arbitrary executable. The trainer will
673 take parameters from your network JSON "Config" block and call your
674 implementation with the equivalent command line. Construction of the
675 command line is controlled by the "CmdlinePrefix" and "CmdlineSep"
676 keywords.

678 <a name="config_sl_output"></a>
679 ## SL Network Output
680 
681 As networks are spawned, the service captures network artifacts in a
682 standard place. This is controlled by the "OutputPath" parameter
683 (which can be separately controlled at the service or module level).
684 Each time the trainer is run, it creates a time-stamped directory
685 under this path, subdirectories for all configured modules beneath
686 that time-stamped directory, and subdirectories for each network
687 beneath those module dirs.
688 
689 Inside each network directory, the trainer places a python script used
690 to start the network (and that can be used to exactly reproduce the
691 network as run by the trainer), and the captured stdout and stderr of
692 the spawned network process.
693 
694 The trainer also creates an output directory inside each network
695 directory where implementations can place output artifacts (e.g.,
696 network weights, convergence metrics, etc). If your config specifies a
697 "ModOutputTag", pointing to a config keywork used by your
698 implementation to set an output directory, the trainer will pass this
699 output directory to your implementation .
700 
701 As an example, suppose your configuration was:
702 
703 ```javascript
704 {
705  # These are all DLTrainer-specific arguments
706 
707  "Name" : "DLAA_hpg18-submission",
708  "Impl" : "imgrst_recurrent_DLTrainer",
709  "OutputPath" : "/raid/data/DLAA/Training/Output",
710  "ModOutputTag" : "folder_path",
711 
712  # These are user-defined (in this case DLAA-specific) arguments
713 
714  "Config" : {
715  "Include" : "Cmdline_Defaults.json",
716  "DLAAConfigs" : "Configs.json",
717  "ActiveConfig" : "msalvi-dgx-wrae",
718  "folder_path" : "/path/to/some/dir"
719  },
720 
721  # Networks to run. These inherit all default parameters above, and override selected parameter values
722 
723  "Networks" : [{"Config" : {"gpu" : 0, "CROP_SIZE" : 128, "BATCH_SIZE" : 2}},
724  {"Config" : {"gpu" : 1, "CROP_SIZE" : 128, "BATCH_SIZE" : 4}},
725  {"Config" : {"gpu" : 2, "CROP_SIZE" : 128, "BATCH_SIZE" : 8}},
726  {"Config" : {"gpu" : 3, "CROP_SIZE" : 192, "BATCH_SIZE" : 2}}]
727 }
728 ```
729 
730 Then DLTrainer would create a directory structure that looks like:

734 ngvpn01-173-211.dyn.scz.us.nvidia.com:Output:>tree -L 4
735 |-- 2018-05-01T15_50_05.446925
736  |-- DLAA_hpg18-submission
737  |-- Network_0
738  | |-- output
739  | |-- script.py
740  | |-- stderr.txt
741  | |-- stdout.txt
742  |-- Network_1
743  | |-- output
744  | |-- script.py
745  | |-- stderr.txt
746  | |-- stdout.txt
747  |-- Network_2
748  | |-- output
749  | |-- script.py
750  | |-- stderr.txt
751  | |-- stdout.txt

769 <a name=sl_coordination></a>
770 ## Coordinating SL Networks
771 
772 * **Timed Runs**
773 
774  A network can be run for a fixed length of time, by specifying the
775  "TimeoutSec" parameter (which you can specify separately at the
776  service-level, module-level or network-level). The SL service
777  will stop the network when the specified number of seconds has
778  elapsed.
779 
780 * **Delayed Runs**
781 
782  You can insert a configurable delay after starting a network, by
783  using the "DelaySec" parameter. This can be useful when spawning
784  multiple networks on a multi-GPU machine, where the network
785  implementation relies on GPU utilization to implicitly select a
786  GPU.
787 
788 * **Scheduling Runs**
789 
790  The SL service provides a simple scheduling mechanism, for
791  configuring many timed networks on a single node, where the number
792  of desired networks may exceed the GPU resources of that node.
793 
794  Parameters controlling the scheduler include:
795 
796  * "RequiresGpu"
797 
798  This should be set to `true` if a network requires
799  GPU resources
800 
801  * "GpuTag"
802 
803  This parameter tells the trainer which module-specific JSON
804  tag specified the GPUs a network wants to use. This is
805  required because there is no standard for specifying GPUs,
806  some experiments using "gpu", others using "gpus".
807 
808  The trainer does however expect either valid GPU indices, or
809  the value `-1` to indicate that any GPU is ok. For example:
810 
811  ```
812  "gpu" : [0, 1]
813  "gpu" : -1
814  "gpu" : [0, 1, 2, -1]
815  ```
816 
817  * "MaxGpuNetworks"
818 
819  This parameter specifies the maximum number of GPU networks
820  that can be run simultaneously. The trainer will take the
821  smaller of this or the number of GPUs actually present, to
822  determine the max allowable number of networks.
823 
824  * "MaxNetworks"
825 
826  This parameter specifies the maximum number of any network
827  (GPU or CPU-based) that can be run simultaneously.
828 
829  When the service starts, the scheduler will spawn whatever
830  configured networks it can, up to the network or resource limits,
831  queuing the remainder to be run when resources are available. As
832  timed runs complete, the scheduler will iterate through the queue
833  of waiting networks, matching GPU requirements with available
834  resources.

850 <a name=config_rl_dltrainer></a>
851 ## DLTrainer Configuration Files
852 
853 As noted above, the `trainerconfig` commandline argument (or the
854 default `TrainerConfig` parameter) specifies a configuration file for
855 DLTrainer itself. This is used to configure how DLTrainer
856 communicates with the broker, what sims the trainer supports, and any
857 behavioral parameters for the trainer itself.
858 
859 For example, suppose the default config/DLTrainer/Trainer_Config.json file
860 contains:
861 
862 ```
863 {
864  "RL" :
865  {
866  "Modules" : ["../simpleTrainingApp/Trainer_Config.json", "../aleApp/Trainer_Config.json"]
867  }
868 }
869 ```
870 
871 This says that the trainer supports data from two different types of
872 DLEnvironment sims, whose configurations are given in
873 `config/simpleTrainingApp/Trainer_Config.json` and
874 `config/aleApp/Trainer_Config.json`, respectively.
875 
876 Absolute paths can be used, and support any exported environment
877 variables. Any time a relative path to a json file is encountered
878 while parsing another json file, the path is taken to be relative to
879 the directory of the immediate parent configuration file.
880 
881 As noted above, the default value for any valid trainer parameter can
882 be overridden by specifying it in the trainer config file, so:
883 
884 
885 ```
886 {
887  "RL" :
888  {
889  "BrokerAddress" : "192.168.0.1:9505",
890  "Modules" : ["../simpleTrainingApp/Trainer_Config.json", "../aleApp/Trainer_Config.json"]
891  }
892 }
893 ```
894 
895 would cause the trainer to contact the broker at IP address
896 192.168.0.1 instead of the default value.
897 
898 The `"Modules"` keyword is required, and specifies a list of simulation
899 types that can be processed by the trainer. Note that this keyword
900 can be a list of JSON files containing the sim configurations (as
901 above), or can instead contain the _contents_ of those files, so the
902 following is also a valid config file:
903 
904 ```
905 {
906  "RL" :
907  {
908  "Modules" :
909  [
910  {
911  "Name" : "SimpleTrainingApp",
912  "Config" : "../simpleTrainingApp/Training_Service.json",
913  "Batchsize" : 10,
914 
915  "Networks" : {
916  "default" : ["staImpl1", "staImpl2"],
917  "special" : "staTestImpl"
918  }
919  },
920 
921  {
922  "Name" : "AleApp",
923  "Config" : "../aleApp/Pong_Training_Service.json",
924  "Batchsize" : 2,
925 
926  "Networks" : {
927  "default" : ["aleImpl1", "aleImpl2"],
928  "special" : "aleTestImpl"
929  }
930  }
931  ]
932  }
933 }
934 ```
935 
936 For each simulation type handled by the trainer, you must specify a
937 name for the simulation type, via the `Name` parameter. This tag
938 should match the name published by the corresponding DLEnvironment
939 sims, and is used by the broker to route data between sims and
940 trainers.
941 
942 For each simulation type, you must also specify a configuration file,
943 via the `Config` parameter. This configuration will be sent to the
944 DLEnvironment sim by the trainer whenever it initiates a new
945 connection.
946 
947 Defaults for any other simulation-type-specific parameters in
948 config/DLTrainer/Parameter_Defaults.json can be overridden by
949 specifying them within the JSON block for that sim type. In the
950 example above, `Batchsize` is 10 for sims of type SimpleTrainingApp, but
951 2 for typ AleApp.
952 
953 If a `Networks` block occurs in the JSON for a simulation type, it
954 controls how networks are deployed for that DLEnvironment type (more
955 on that below at <a href=#impls>Network Implementations</a>). If no
956 networks are defined, the default network implementation is taken from
957 the `DefaultImpl` parameter.
958 
959 For details on interpreting DLEnvironment configuration files, see below.
960 
961 <a name=config_rl_dlenv></a>
962 ## DLEnvironment Configuration Files
963 (<a href=#contents>Back to Contents</a>)
964 
965 As detailed above, each DLEnvironment block in a trainer config file
966 must specify a configuration file that will be sent to the sim whever
967 it initiates a connection to the trainer, via the `Config` keyword.
968 
969 For example, the Trainer_Config.json file in the example above
970 specifies that the contents of
971 `config/simpleTrainingApp/Training_Service.json` should be sent to
972 simulations of type SimpleTrainingApp when they connect. That file
973 might look like:
974 
975 ```
976 {
977  "StateOutput": "StateOutput",
978  "StatisticsOutput": "Statistics_Service",
979 
980  "AIInstances": [
981  "Training_AIInstance.json"
982  ],
983 
984  "Experiment": "Experiment.json",
985 
986  "ApplicationSettings": {
987  "AreaSize": 5.0,
988  "Speed": 0.5,
989  "Distance": 3.0
990  }
991 }
992 ```
993 
994 This defines the number of AI Instances (in this case one) and points
995 to a configuration file for each instance in the list (in this case
996 Training_AIInstance.json).
997 
998 When a DLEnvironment sim connects to the trainer (via the broker or
999 the REST interface), the trainer sends its configuration (in the
1000 example above, `config/simpleTrainingApp/Training_Service.json`) down to
1001 the sim.
1002 
1003 When sent by the trainer, any JSON files referenced within the
1004 DLEnvironment config file hierarchy are replaced by their contents.
1005 
1006 The config supports running multiple networks, and the json has been
1007 extended to allow things like:
1008 
1009 ```
1010 {
1011  "StateOutput": "StateOutput",
1012  "StatisticsOutput": "Statistics_Service",
1013 
1014  "AIInstances": [
1015 
1016  {
1017  "Config":"Training_AIInstance.json",
1018  "Network": "default"
1019  },
1020 
1021  {
1022  "Config":"Training_AIInstance.json",
1023  "Network": "special"
1024  },
1025 
1026  ],
1027 
1028  "Experiment": "Simple_Experiment.json",
1029 
1030  "ApplicationSettings": {
1031  "AreaSize": 5.0,
1032  "Speed": 0.5,
1033  "Distance": 3.0
1034  }
1035 }
1036 ```
1037 
1038 Note that each element of the AIInstance array has been replaced with
1039 JSON which specifies the config for that instance, as well as a
1040 network to process data for that instance. These network names are
1041 labels that must defined in the trainer config via a block of commands
1042 like:
1043 
1044 ```
1045  "Networks": {
1046  "default": "experimentImpl",
1047  "special": "specialImpl"
1048  },
1049 ```
1050 
1051 The example above tells the trainer that two different network
1052 implementations may be used (and defines the python modules to use for
1053 the implementations), declares two AI instances, and allows the
1054 configuration to associate each AI instance with a different network.
1055 On loading the configuration file, the trainer will spawn network
1056 objects for any network in the list that is actually referenced by an
1057 AI configuration.
1058 
1059 Any AI instance that doesn't specify a network will be connected to
1060 the default network. If a `Networks` block is defined in the trainer
1061 config file, this will be the network labeled `default`. If no
1062 `Networks` block is present, or the block doesn't define a default,
1063 then this will be the network specified by the `DefaultImpl` parameter
1064 (see discussion of default parameter handling starting at <a
1065 href=#trainer_defaults>Default Parameter Values</a> above).

1069 ## Network Implementations
1070 
1071 (<a href=#contents>Back to Contents</a>)
1072 
1073 DLTrainer is a famework for deploying network implementations against
1074 DLEnvironment data.
1075 
1076 It allows you to:
1077 
1078 * deploy a single default network to process all DLEnvironment data
1079 
1080 * deploy multiple instances of the same network that DLEnvironment
1081  data of the same type will be distributed among
1082 
1083 * deploy different network implementations that data for the same sim
1084  type will be distributed among
1085 
1086 * deploy different network implementations for different AI instances
1087  in the same DLEnvironment sim
1088 
1089 * deploy different network implementations for different types of
1090  DLEnvironment sim data
1091 
1092 or all of the above. These are explained in more detail below.
1093 
1094 The name of the default network implementation module to deploy can be
1095 specified via configuration files, as discussed above, or via
1096 commandline arguments when starting the trainer. For example:
1097 
1098 ```
1099 python DLTrainer.py defaultimpl=experimentImpl
1100 ```
1101 
1102 would load experimentImpl.py as the default network implementation.
1103 Network implementations known to DLTrainer are contained under the
1104 `impls` directory, but network implementations can live anywhere in your
1105 PYTHONPATH path.
1106 
1107 Each network implementation must be self-contained (any module it
1108 depends on must be loaded by the implementation), and must export the
1109 following functions:
1110 
1111 * `getActionArgs()`
1112 
1113  This should return a python list of arguments. The length of the
1114  list determines how many copies of the network implementation the
1115  trainer will run, and the ith element of the list is passed to the
1116  ith copy of the network every time takeAction(arg, state) is
1117  called (on a tick from a client - see below)
1118 
1119 * `takeAction(args, state)`
1120 
1121  This is called on every tick from a client, with the
1122  initialization arguments (if any) for this network, and the current
1123  state of the client sim. It should return the actions to be sent
1124  back to the client from the training network.
1125 
1126 Additionally, you can control what constitutes an episode and
1127 experiment by defining the optional functions:
1128 
1129 * `checkEpisodeComplete(args, state, numTicks, ticksPerEpisode)`
1130 
1131  This is called on every tick to determine if the current episode has
1132  finished. The function is passed the initialization arguments (if
1133  any) for this network, the current state, the number of ticks in the
1134  current episode, and the configured number of ticks per episode (from
1135  the `TicksPerEpisode` parameter for this simulation type).
1136 
1137  If your implementation doesn't define this function, an episode is
1138  considered complete when numTicks == ticksPerEpisode.
1139 
1140 * `checkExperimentComplete(args, state, numEpisodes, episodesPerExperiment)`
1141 
1142  This is called whenever an episode finishes to determine if the
1143  current experiment has finished. The function is passed the
1144  initialization arguments (if any) for this network, the current
1145  state, the number of episodes in the current experiment, and the
1146  configured number of episodes per experiment (from the
1147  `EpisodesPerExperiment` parameter for this simulation type).
1148 
1149  If your implementation doesn't define this function, an experiment is
1150  considered complete when numEpisodes == episodesPerExperiment.
